# Frontend Internationalization (i18n)

## Overview

The frontend ships a lightweight, hand-rolled i18n layer instead of pulling in
a heavy framework — consistent with the project's existing style of small
custom utilities (`Router`, `GenericClient`, `AuthEvents`, etc.). Today only
English (`en`) is bundled, but the `Translator` API, language persistence,
and the header language selector are fully built out so adding a new
language is a drop-in change (see [Adding a new language](#adding-a-new-language)
below).

## Where translations live

Translation strings live in YAML files under `frontend/assets/i18n/`, one
file per language:

```
frontend/assets/i18n/
  en.yaml
```

Each file is bundled at build time via Vite's `?raw` import suffix (no
runtime fetch) and parsed with [`js-yaml`](https://github.com/nodeca/js-yaml)
when `frontend/assets/js/i18n/Translator.js` is first loaded.

## The `namespace.key` convention

Keys are grouped under one top-level namespace per page or element, matching
the existing folder layout under `components/elements` and `components/pages`.
For example:

```yaml
header:
  title: Majora
  login: Login
login_modal:
  title: Login
  username_label: Username
character_info:
  class_label: 'Class:'
  level_label: Level
```

This keeps keys easy to locate: if you're editing `HeaderHelper.jsx`, its
strings live under the `header:` namespace in `en.yaml`.

## Using `Translator.t()` in components

`frontend/assets/js/i18n/Translator.js` exposes a static `t(key, fallback)`
method that performs a dot-path lookup against the current language's map:

```js
import Translator from '../../../i18n/Translator.js';

Translator.t('header.login'); // => 'Login'
Translator.t('missing.key');  // => 'missing.key' (the key itself, as a fallback)
Translator.t('missing.key', 'N/A'); // => 'N/A'
```

If a key is missing, `t()` returns the fallback (or the key itself when no
fallback is given) instead of throwing — a typo in a key surfaces as
visibly-wrong text in the UI rather than crashing the render.

Only user-visible text is translated. `data-testid` attributes, console/log
messages, and code comments are left as-is.

## Language selector and persistence

`frontend/assets/js/components/elements/LanguageSelector.jsx` (+ its
`helpers/LanguageSelectorHelper.jsx` and
`controllers/LanguageSelectorController.js`, following the project's
Component/Helper/Controller trio convention) renders a dropdown in the
header showing a flag emoji and language code per registered language (e.g.
"🇬🇧 en"). Selecting an entry calls `Translator.setLanguage(code)`, which:

1. Updates the in-memory current language used by `Translator.t()`.
2. Persists the choice via `frontend/assets/js/i18n/LanguageStorage.js`
   (a thin `localStorage` wrapper guarding against unavailable
   `localStorage`, mirroring `AuthStorage.js`).
3. Emits a `language:changed` `window` event via
   `frontend/assets/js/i18n/LanguageEvents.js` (mirroring `AuthEvents.js`).

`Translator` reads the persisted language from `LanguageStorage` on module
load, so a page reload keeps the last choice.

## How the UI reacts to a language change

`App.jsx` keeps a `lang` state initialized from `Translator.getLanguage()`
and subscribes to `LanguageEvents` (wired through `AppController.buildEffect()`,
alongside the existing `hashchange` listener). `AppHelper.render` extends its
existing `key={hash}` remount trick to `key={`${hash}:${lang}`}\`, so every
page/element under the `React.Fragment` re-renders with the new language on
selection — without requiring every component to individually subscribe to
translation updates.

## Adding a new language

1. Add a new `<code>.yaml` file under `frontend/assets/i18n/` (e.g. `pt.yaml`),
   mirroring every key already present in `en.yaml`.
2. Import the new file in `Translator.js` and register it in the
   `TRANSLATIONS` lookup table:
   ```js
   import ptYaml from '../../i18n/pt.yaml?raw';
   // ...
   const TRANSLATIONS = {
     en: load(enYaml),
     pt: load(ptYaml),
   };
   ```
3. Add a flag entry for the new code in
   `LanguageSelectorController`'s `FLAGS` table so it shows up correctly in
   the dropdown.

No other code changes are required — `Translator.getAvailableLanguages()`
and the selector automatically pick up the new entry.
